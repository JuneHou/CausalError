# Causal Intervention Pipeline — Session Notes

Reference: full pipeline design → `benchmarking/causal/patch/WORKFLOW.md`

---

## What This Pipeline Does

Validates causal graph edges A→B from `data/trail_causal_outputs_AIC/capri_graph.json`
(12 AIC-pruned edges) using do(A=0) counterfactual intervention on GAIA agent traces.

Fix error A at span t_A → rerun agent → measure whether downstream error B disappears.

---

## Hard Points Resolved This Session

### 1. Deduplication: AInstanceRecord + EdgePair split

**Problem**: Original pipeline ran patch/rerun/Judge-A once per (A-instance × B-type),
causing redundant LLM calls for the same A span.

**Solution**: Two dataclasses:
- `AInstanceRecord` — one per unique A error_id; drives patch, rerun, Judge-A
- `EdgePair` — one per (error_id × B-type); drives Judge-B fan-out only

All EdgePairs sharing the same `error_id` reuse one `RerunResult`.
Files: `case_builder.py` → `a_instances.jsonl` + `edge_pairs.jsonl`

---

### 2. Real LLM rerun (no proxy mode)

**Problem**: Original rerun used a "proxy mode" that wrote empty suffix spans,
so Judge-B had no counterfactual trace to compare against.

**Solution**: Real LLM rerun via `litellm`. Key sub-problems solved:

**smolagents message format**: TRAIL traces use non-standard roles:
- `tool-call` → must convert to OpenAI `assistant` + `tool_calls`
- `tool-response` → must convert to OpenAI `tool` + `tool_call_id`
- Tool call `arguments` are stored as Python dicts → must serialize to JSON strings

**Tool result replay**: Live tools (web browser etc.) unavailable.
Original TOOL spans after t_A are indexed by tool name in timestamp order and replayed
in sequence during the LLM continuation loop.

**patch_side**:
- `replace_span_output` (Formatting Errors): inject patched assistant message directly,
  add replayed tool results for any tool calls in the patch, then loop
- `replace_span_input` (most others): replace last user message, re-call LLM at t_A,
  then loop

---

### 3. Rerun loop termination bug

**Problem**: The continuation loop had two incorrect early exits:
1. `if last_role == "assistant" and not tool_calls: break` at loop start —
   killed the loop immediately for any patched message without a tool call
   (common for `replace_span_input` patches that produce a planning step)
2. `if not tc_list: break` after each LLM call —
   stopped at the first plain-text (planning/thinking) LLM response,
   even though those are valid intermediate steps before further tool calls

**Fix** (`rerun_harness.py`, `_real_rerun` loop):
- Remove both early exits
- Loop always calls LLM once per iteration (each call = one step)
- Only stop on `final_answer` tool call or `max_steps_after` exhausted
- Plain-text responses (no tool calls) continue to next iteration

---

### 4. max_steps_after unit and value

**Problem**: Previous analysis used `span_rank` distances (counts all span types).
`max_steps_after` counts **LLM calls** (one per loop iteration), not all spans.

**Analysis**: Computed proper LLM-step distances across 116 GAIA traces.
Trace structure: `ToolCallingAgent` children = CHAIN steps (LLM+TOOL) + direct LLM calls.
Each is one step regardless of type.

**Results** (LLM steps from A to B):
```
p50=3  p75=6  p90=10  p95=13  max=26

max_steps_after=8   → 87% coverage
max_steps_after=12  → 94% coverage  ← new default
max_steps_after=15  → 96% coverage
```

**Fix**: Changed default from 8 → 12 in `rerun_harness.py` and `run_pipeline.py`.

---

### 5. Split model: rerun uses o3-mini, everything else uses gpt-4o

**Reason**: The GAIA error logs were generated by o3-mini. The counterfactual rerun
must use the same model to produce a realistic continuation. Patch generation and
judges (Judge-A, Judge-B) use gpt-4o.

**Fix**: Added `--rerun_model` argument to `run_pipeline.py` (default `openai/o3-mini`),
separate from `--model` (default `openai/gpt-4o`). `step3_rerun` passes `rerun_model`
to `_rh.run_rerun()`. Default in `rerun_harness.run_rerun()` also changed to `o3-mini`.

`o3-mini` is already handled correctly in `_call_messages`: the `"o3" in model` branch
drops `temperature` and sets `reasoning_effort="medium"`.

---

### 6. Missing "emerged" label in Judge-B

**Problem**: Label set had no option for B absent in baseline but present in rerun.
The LLM would fall back to `unchanged`, which is semantically wrong — the intervention
*introduced* a new error B that wasn't there before.

**Fix** (`judge_b_effect.py`):
- Added `"emerged"` to `VALID_EFFECT_LABELS`, `JUDGE_B_SYSTEM`, and the output schema.
- Definition: B was ABSENT in baseline, NOW PRESENT in rerun (intervention introduced B).
- `not_observable` is now strictly: B absent in baseline AND absent in rerun.

Full label semantics:
```
disappeared   B present baseline → absent rerun      (good: intervention helped)
delayed       B present baseline → present rerun, later onset
unchanged     B present baseline → present rerun, same onset
earlier       B present baseline → present rerun, earlier onset
weakened      B present baseline → present rerun, lower severity
strengthened  B present baseline → present rerun, higher severity
emerged       B absent  baseline → present rerun      (bad: intervention introduced B)
not_observable B absent baseline → absent rerun       (no causal signal either way)
```

---

### 7. effect_aggregator.py key mismatch

**Problem**: After removing `edge` from `AResolvedVerdict`, the aggregator broke
with `KeyError: 'edge'` because it tried to key `resolved_idx` on `(trace_id, error_id, edge)`.

**Fix**: `resolved_idx` keyed on `(trace_id, error_id)` only.
Patch failure rate uses `template_used` field (not `edge.a`).

## Session 2 — Fixes (test_run pipeline debugging + span parsing)

### 8. REQUIRED_MARKERS postcheck bug

**Problem**: Entire comma-separated string was checked as one substring.
e.g. `"<end_plan>, <end_code>"` was searched literally, never matching two separate markers.

**Fix** (`patch_generator.py`, `_normalize_required_markers()`):
Split on `,` and strip each token before checking each marker individually.

---

### 9. TOOL→LLM span remap for tool call arguments

**Problem**: Tool call arguments live in the parent LLM span's `output.tool_calls`, not in
the TOOL span itself. Annotating the TOOL span and patching `replace_span_input` was wrong —
the arguments are authored by the LLM's output, not the tool's input.

**Fix** (`case_builder.py`, `_find_sibling_llm_span()`):
When `annotated_span_kind == "TOOL"` and `patch_side == "replace_span_input"`,
walk to parent and find the sibling LLM span, then flip `patch_side` to `replace_span_output`.
Both `annotated_location` and `intervention_location` are preserved for audit.
Added `annotated_span_kind` and `intervention_span_kind` fields to `AInstanceRecord`.

---

### 10. Resource Abuse postcheck — hallucinated tool_call keys

**Problem**: LLM patch generator added phantom keys (`stop_criterion`, `stop_condition`)
to the tool_calls JSON when patching Resource Abuse tool call spans, which doesn't match
the OpenAI schema and breaks the rerun.

**Fix A** (`patch_generator.py`): Added `snippet_mode="tool_call_json"` gate.
When `annotated_span_kind == "TOOL"` and `patch_side == "replace_span_output"`,
postcheck validates that no novel keys beyond `{id, type, function}` appear in tool_call entries.

**Fix B** (`patch_library.json`): Extended Resource Abuse `error_type_spec_text` with
`output_side_note` explaining the tool_calls JSON case — modify ARGUMENTS (query, remove
filter_year), not add phantom fields; only valid keys are `id`, `type`, `function`.

---

### 11. Judge-A template differentiation (replace_span_output vs replace_span_input)

**Problem**: Single Judge-A template was used for both patch sides. For `replace_span_input`,
the judge looked for correct output behavior in the patched input — penalizing valid patches
because an input shouldn't contain output behavior.

**Fix C** (`judge_a_resolved.py`): Two separate user templates:
- `JUDGE_A_USER_OUTPUT_TEMPLATE` for `replace_span_output`: compare ORIGINAL_SPAN vs PATCHED_SPAN
- `JUDGE_A_USER_INPUT_TEMPLATE` for `replace_span_input`: check that error-causing context
  was removed/corrected, with explicit NOTE not to require correct output behavior in an input.

**Fix D** (`judge_a_resolved.py`, SYSTEM prompt): Demoted RERUN_SUFFIX to "supplementary
context only — do not use to determine RESOLVED/UNRESOLVED". Primary verdict must come from
ORIGINAL_SPAN vs PATCHED_SPAN comparison alone.

---

### 12. Shared intervention_location overwrite (Fix E)

**Problem**: Multiple A-instances pointing to the same intervention span would overwrite
each other's patch in the rerun, making results non-reproducible and changing the causal estimand.

**Fix E** (`case_builder.py`, `dedup_by_intervention_location()`):
When multiple A-instances share the same `intervention_location`, keep the first by
`annotation_index` and write the rest to `intervention_location_conflicts.jsonl`.
Each conflict record includes `kept_error_id` for traceability.

---

### 13. Agent-steps span definition (span_level_parser.py)

**Problem**: Original `build_step_spans()` returned only level-1 children of `main` span
(always 2 per trace: `get_examples_to_answer` + `answer_single_question`), giving avg 3.0
steps/trace — far from Table 5's 8.28 avg (118 traces, 977 total).

**Investigation**: GAIA traces use nested two-level agent structure:
`main → answer_single_question → CodeAgent.run → Step N → LLM/TOOL`.
Agent-steps = `CodeAgent.run` + `ToolCallingAgent.run` + `Step N` spans at any depth.
This gives 980 total for 117 traces (avg 8.38) — matches Table 5's 977 (8.28) within +3
(accounting for 1 missing trace with 626 non-recursive spans: `117×3 + 626 = 977`).

**Fix** (`span_level_parser.py` rewritten; original saved as `span_level_parser_old.py`):
- `build_step_spans()`: recursively collects all `CodeAgent.run`, `ToolCallingAgent.run`,
  and `Step N` spans at any depth under `main_span` (not just level-1 children)
- `annotated_span_to_step()`: walks `parent_of` up to nearest ancestor in step_span_set
  (nearest `Step N` or runner span), not level-1 under main
- Interface unchanged — `trail_io.py` imports same functions without modification

---

### 14. Tool Selection Errors — input-side patch quality (no tool call in LLM output)

**Problem**: TSE patch spec said "rewrite local step to use correct tool" — for input-side
patches, LLM added structured context (FACTS LIST) rather than a tool-use directive,
leaving the core issue (agent answered from memory) unaddressed. Judge-A correctly
identified the tool was still not invoked.

**Fix** (`patch_library.json`, Tool Selection Errors `error_type_spec_text`):
Repair instruction now explicitly says: if agent answered from memory without calling any tool,
append a short mandatory instruction to the last user/system message —
`"IMPORTANT: You MUST verify this using [CORRECT_TOOL_HINT] before providing a final answer."`
Postcheck extended: patch payload must contain an explicit reference to `CORRECT_TOOL_HINT`.

**Fix** (`patch_generator.py`, TSE postcheck rule):
Added check: if `CORRECT_TOOL_HINT` is filled but absent from `patch_payload`, fail with
message directing the LLM to include it explicitly.

**Result**: `2713fa0a TSE|1` resolved (conf=1.0) after this fix.

## Trace Structure (GAIA)

```
main
└── answer_single_question
    └── ToolCallingAgent.run   ← agent span; its direct children = "steps"
        ├── LiteLLMModel.__call__   (pre-loop LLM call, no tool)
        ├── Step 1 [CHAIN]
        │   ├── LiteLLMModel.__call__   ← annotation location points here
        │   └── SomeTool [TOOL]
        ├── LiteLLMModel.__call__   (standalone planning, no tool — valid step)
        ├── Step 2 [CHAIN]
        │   └── ...
        └── ...
```

Annotations `location` field = `LiteLLMModel.__call__` span_id inside a Step.
`span_rank` in `trail_span_order` covers ALL span types (LLM + CHAIN + TOOL).
`max_steps_after` counts only LLM calls (direct children of ToolCallingAgent).
